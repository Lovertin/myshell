根据课程设计要求文档，**实验一 Shell编程**的主要目标是编写一个C语言程序，作为Linux内核的命令行解释程序（Shell） 。你需要实现接受命令、解释命令、采用**创建子进程**的方式执行程序，以及处理父子进程同步等核心逻辑 。

以下是进行实验一的整体设计与实现步骤指南：

### 一、 明确实验任务与功能要求

你的C语言程序需要支持以下核心功能 ：

1. **基础命令：** `ls`、`cd`、`cat`、`grep`、`echo`、`type` 。

2. **高级特性：** 
* **查阅历史记录 (`history`)** ：记录用户输入的命令，并支持通过 `!n` 运行指定历史命令 。

* **别名功能 (`alias`/`unalias`)** ：允许用户自定义简单字符串来替换复杂命令 。

* **命令补全** ：支持按下 `Tab` 键自动补全命令或文件名 。

* **后台运行** ：在命令后加 `&`，使程序在后台异步运行，不阻塞Shell 。

* **管道 (`|`) 与重定向 (`>`、`<`)** ：实现多个命令的数据传输与标准输入输出重定向 。

---

### 二、 核心主循环架构设计

Shell 的本质是一个死循环，其基本执行流程如下：

```c
while(1) {
    // 1. 打印提示符（如 "username@hostname:current_dir$ "）
    print_prompt(); 
    
    // 2. 读取用户输入的命令字符串（需考虑实现Tab键补全，可能要用到 readline 库）
    char *cmd_line = read_command(); 
    
    // 3. 将命令加入历史记录
    add_to_history(cmd_line); 
    
    // 4. 解析命令字符串（切分成命令名、选项、参数，处理别名替换）
    Command cmd = parse_command(cmd_line); 
    
    // 5. 执行命令
    execute_command(cmd);
}

```

### 三、 关键技术的具体实现指导

#### 1. 内建命令 vs 外部命令

* **内建命令 (Built-in Commands)：** 诸如 `cd`、`exit`、`alias`、`history` 等命令，不能通过创建子进程来执行（因为子进程无法改变父进程的当前工作目录或变量）。这些命令需要在主进程中直接用C语言逻辑或系统调用实现（例如 `cd` 对应调用 `chdir()` 函数）。
* **外部命令：** 诸如 `ls`、`cat`、`grep` 等。通过 `fork()` 创建子进程，在子进程中调用 `execvp()` 或 `execv()` 等系统调用函数去执行对应的系统二进制文件 。



#### 2. 命令执行与进程控制（`fork` + `exec` + `wait`）

对于外部命令，核心逻辑为：

* **父进程：** 调用 `fork()` 。如果命令末尾**没有** `&`，则调用 `wait()` 或 `waitpid()` 阻塞等待子进程结束 ；如果有 `&`，则直接打印子进程 PID 并进入下一轮循环（后台运行） 。


* **子进程：** 负责处理重定向和管道，然后调用 `execvp(args[0], args)` 执行程序。

#### 3. 管道与 I/O 重定向的实现

* **输入/输出重定向 (`<`, `>`)：**
* 在子进程中，先解析出是否含有 `>` 或 `<` 符号及对应的文件名。
* 使用 `open()` 打开目标文件，再利用 `dup2(fd, STDOUT_FILENO)`（针对 `>`）或 `dup2(fd, STDIN_FILENO)`（针对 `<`）将标准输入输出重定向到文件。


* **管道 (`|`)：**
* 如果解析出有多段命令由 `|` 连接，需要利用 `pipe(pipefd)` 创建管道。
* 多次 `fork()` 创建多个子进程，前一个子进程将标准输出重定向到管道的写端（`dup2(pipefd[1], STDOUT_FILENO)`），后一个子进程将标准输入重定向到管道的读端（`dup2(pipefd[0], STDIN_FILENO)`）。



#### 4. 高级功能实现技巧

* **命令补全：** 在Linux下实现交互式按 `Tab` 补全，推荐借助 GNU 的 `readline` 库（如 `readline()` 和 `rl_attempted_completion_function`），它可以极大简化捕捉 `Tab` 键并搜索当前路径下文件及系统环境变量匹配项的开发难度 。


* **别名功能 (`alias`)：** 可以使用一个链表或结构体数组存储别名映射对 `(alias_name, real_command)` 。在解析命令时，优先在别名表中查找，如果匹配则将命令替换为完整的真实命令字符串再处理 。


* **历史记录 (`history`)：** 在内存中开辟一个循环数组或链表记录最近输入的 `HISTSIZE` 条命令 。在遇到 `!n` 输入时，从数组中取出第 `n` 条命令重新走解析与执行流程 。

### 四、 实验开发建议

1. **环境准备：** 推荐在 Ubuntu 或欧拉等 Linux 虚拟机构建的开发环境中进行 ，安装好 `gcc`、`make` 以及可能需要的 `libreadline-dev` 库。


2. **逐步递进开发：**
* **第一阶段：** 先实现能打印提示符，并成功执行最基础的外部单条命令（如 `ls`） ；


* **第二阶段：** 加入内建命令支持（`cd`, `exit`）及别名、历史记录功能 ；


* **第三阶段：** 攻克重定向、后台运行与单管道 ；


* **第四阶段（冲刺高分）：** 完善命令补全和多级管道 。


3. **注意细节：** 在进程退出或创建失败时要妥善回收资源，避免僵尸进程的产生，并处理好各种异常输入的报错提示。
